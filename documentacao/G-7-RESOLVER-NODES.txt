BBP-0007 — Resolver Nodes
Status
Draft
Título
Nós de Resolver da Rede BBP
Resumo
Resolver Nodes são nós de rede responsáveis por executar resolvers BBP e fornecer serviços de resolução de ativos para aplicações e usuários.
Esses nós operam dentro de uma rede descentralizada peer-to-peer (P2P) e consultam dados de blockchains através de indexadores públicos ou endpoints RPC.
A rede de Resolver Nodes permite que múltiplos participantes forneçam serviços de resolução sem depender de infraestrutura centralizada.
Resolvers permanecem stateless e read-only, realizando apenas leitura e agregação de dados provenientes de redes externas.
________________________________________
1. Visão Geral
Resolver Nodes são instâncias de software que executam o mecanismo de resolução BBP dentro de uma rede distribuída.
Eles permitem que aplicações consultem ativos BBP sem precisar implementar resolvers localmente.
Arquitetura simplificada:
Client Application
↓
Resolver Node
↓
Public Indexers / RPC
↓
Blockchains
Múltiplos nós podem existir simultaneamente na rede.
Cada nó pode resolver identificadores BBP de forma independente.
________________________________________
2. Funções de um Resolver Node
Um Resolver Node executa as seguintes funções:
1.	Receber requisições de resolução BBP
2.	Interpretar identificadores BBP
3.	Consultar o Registry BBP
4.	Recuperar manifestos de ativos
5.	Consultar dados em redes blockchain
6.	Retornar resultados de resolução para clientes
Opcionalmente, um nó também pode:
•	manter cache temporário de resoluções
•	encaminhar consultas para outros nós
•	fornecer APIs públicas para aplicações
________________________________________
3. Arquitetura da Rede P2P
Resolver Nodes operam em uma rede peer-to-peer.
Essa rede permite:
•	descoberta de nós
•	compartilhamento de consultas
•	distribuição de carga
•	redundância de resolução
Arquitetura conceitual:
Resolver Node
 ↕
Resolver Node
 ↕
Resolver Node
Cada nó pode atuar como:
•	cliente de resolução
•	provedor de resolução
•	relay de consultas
________________________________________
4. Descoberta de Nós
Para participar da rede, um Resolver Node precisa descobrir outros nós.
Isso pode ocorrer através de mecanismos como:
•	bootstrap nodes
•	listas públicas de peers
•	tabelas distribuídas (DHT)
Processo típico:
Node start
↓
connect to bootstrap nodes
↓
discover peers
↓
join P2P network
Bootstrap nodes apenas facilitam a entrada inicial na rede e não exercem controle sobre ela.
________________________________________
5. Comunicação entre Nós
Resolver Nodes trocam informações através de mensagens P2P.
Tipos comuns de mensagens incluem:
RESOLVE_REQUEST
RESOLVE_RESPONSE
PEER_DISCOVERY
CACHE_QUERY
Essas mensagens permitem que nós:
•	solicitem resolução de ativos
•	retornem resultados
•	descubram novos peers
•	compartilhem dados em cache
________________________________________
6. Cache Distribuído
Resolver Nodes podem manter caches temporários de resultados de resolução.
Objetivos do cache:
•	reduzir latência
•	diminuir consultas a indexadores externos
•	melhorar resiliência da rede
Exemplo de estratégia:
cache_key = BBP identifier
TTL = configurable
Caches não são considerados fonte autoritativa de estado.
Eles apenas aceleram resoluções repetidas.
________________________________________
7. Integração com Indexadores Públicos
Resolver Nodes consultam dados de blockchains através de:
•	indexadores públicos
•	APIs RPC
•	gateways de rede
Exemplos de infraestrutura pública incluem serviços semelhantes a:
•	Mempool.space
•	Infura
•	Alchemy
Nós podem utilizar múltiplos provedores para redundância e fallback.
A lógica de integração com cada rede é definida em:
BBP-0008 — Network Adapters
________________________________________
8. Rede de Teste
Durante fases iniciais do protocolo, Resolver Nodes podem operar em uma rede simplificada.
Arquitetura típica de teste:
Viewer / App
↓
Local Resolver
↓
Public Indexers
↓
Blockchains
Opcionalmente, aplicações podem compartilhar dados através de redes P2P leves baseadas em navegador.
Esse modelo permite testes do protocolo sem necessidade de infraestrutura dedicada.
________________________________________
9. Rede de Produção
Em uma implementação completa, a rede BBP pode operar com múltiplos Resolver Nodes distribuídos.
Arquitetura:
Client
↓
P2P Resolver Network
↓
Network Adapters
↓
Blockchains
Essa rede permite:
•	alta disponibilidade
•	descentralização
•	múltiplas fontes de resolução
________________________________________
10. Independência dos Nós
Resolver Nodes operam de forma independente.
Eles não participam de:
•	consenso de rede
•	validação de blocos
•	armazenamento de estado global
Sua função é exclusivamente:
retrieve
interpret
resolve
Essa abordagem permite que qualquer participante da rede execute um nó sem necessidade de coordenação central.
________________________________________
11. Segurança e Confiabilidade
Implementações podem adotar estratégias para melhorar confiabilidade, incluindo:
•	múltiplos indexadores
•	fallback automático
•	verificação de respostas
•	limitação de consultas
Essas estratégias não são obrigatórias, mas são recomendadas para implementações públicas.
________________________________________
12. Papel no Protocolo BBP
Resolver Nodes constituem a camada de execução distribuída do protocolo BBP.
Eles conectam aplicações ao ecossistema de ativos digitais representados por identificadores BBP.
Arquitetura completa:
DNA
↓
Registry
↓
Manifest
↓
Resolver
↓
Resolver Nodes
↓
Network Adapters
↓
Blockchains
________________________________________
Conclusão
Resolver Nodes permitem que o protocolo BBP opere como uma infraestrutura descentralizada de resolução de ativos.
Principais características da rede:
•	descentralizada
•	baseada em leitura de estado
•	compatível com múltiplas blockchains
•	operável por qualquer participante
Essa arquitetura permite que aplicações utilizem o protocolo BBP sem depender de servidores centralizados.
