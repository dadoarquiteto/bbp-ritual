1. Visão Executiva
A Tecnologia Viva é um protocolo de criptografia contextual onde chaves privadas nunca existem em estado permanente. Elas nascem do contexto do momento — ambiente, dispositivo, rede e entropia viva —, são usadas por milissegundos, e então destruídas. A única coisa armazenada é um snapshot determinístico do contexto, suficiente para recriar a chave no futuro sob as mesmas condições verificáveis.
Este documento apresenta a especificação técnica revisada do protocolo, o código real da Living Wallet integrada ao BBP (Bitcoin Blueprint Protocol), e a arquitetura do Oráculo Descentralizado com threshold/MPC — mantendo o espírito de tecnologia viva, onde a rede em si se torna a fonte de segurança.

1.1 Princípios Fundamentais
•	Chave efêmera: a chave privada existe apenas durante o uso — não antes, não depois.
•	Contexto como segredo: a segurança vem do contexto que gerou a chave, não da chave em si.
•	Reconstrução determinística: qualquer contexto registrado pode recriar a chave original.
•	Oráculo distribuído: nenhum nó único controla a entropia — a rede inteira é o oráculo.
•	Tríade inseparável: Oráculo + Setup + Fragmentação. Sem os três juntos, nada abre.

1.2 O que é genuinamente original

Chaves efêmeras (ephemeral keys)	Sim (TLS, Signal)	Nunca com contexto físico vivo
Snapshot determinístico para reconstrução	Parcialmente	Inédito nesta forma
Fragmentação com ordem derivada do oráculo	Não	Original
Setup baseado em identidade de dispositivo	Sim (device binding)	Nunca integrado ao oráculo assim
Oráculo crescendo com a rede (entropia emergente)	Não	Original

2. Arquitetura Geral — As Três Camadas
O protocolo opera em três camadas inseparáveis. Nenhuma funciona sozinha. Um atacante com acesso a qualquer subconjunto de duas camadas continua sem capacidade de reconstruir os dados.

2.1 Camada 1 — Oráculo Vivo
O Oráculo é o mecanismo de entropia e a âncora de reconstrução contextual. Em sua forma inicial, é um servidor que registra eventos. Em sua forma evoluída, é uma rede descentralizada de nós onde nenhum nó individual tem o segredo completo.
•	Coleta entropia de múltiplas fontes simultâneas (beacon, blockchain, hardware, rede).
•	Gera o context_snapshot — o único artefato gravado, suficiente para recriar qualquer chave.
•	Nunca armazena chaves — armazena apenas os ingredientes verificáveis para recriá-las.
•	Janela deslizante de 7 dias: snapshots expiram e ficam matematicamente inacessíveis.

2.2 Camada 2 — Setup (LiveSetupID)
O Setup é a identidade viva do dispositivo e do usuário. É gerado localmente e nunca sai da máquina. O oráculo recebe apenas uma referência (hash), não o setup completo.
•	Combina: machine_id, cpu_temp, jitter de rede, movimento do mouse, user_id, pin secreto.
•	Resultado: fingerprint único por dispositivo — diferente mesmo em máquinas idênticas.
•	Anti-clone: jitter e temperatura nunca se repetem exatamente entre dispositivos diferentes.
•	Janela de 7 dias com renovação automática a cada uso.
Melhoria crítica para produção: substituir MAC/CPU_ID por TPM (Windows) ou Secure Enclave (Apple) — chips de hardware que tornam a clonagem fisicamente impossível, não apenas computacionalmente difícil.

2.3 Camada 3 — Fragmentação (LiveDataFractal)
Os dados nunca são armazenados inteiros. São fragmentados em N partes com ordem determinada pelo oráculo. A ordem de montagem só pode ser recalculada com o snapshot + setup corretos.
•	Fragmentos individuais não têm significado — são lixo sem a ordem correta.
•	Ordem derivada via Fisher-Yates com seed determinística do snapshot + setup.
•	Hash individual por fragmento: qualquer adulteração é detectada antes da remontagem.
•	Fragmentos falsos podem ser inseridos para confundir atacantes.

2.4 Fluxo Completo — Criptografar e Descriptografar
Fluxo de criptografia (emissão):
Evento acontece (transação, abertura de arquivo, mensagem)
  → Oráculo coleta entropia (beacon + chain + hardware + rede)
  → Oráculo gera context_snapshot (assinado, verificável)
  → Setup + snapshot → KDF → chave efêmera (32 bytes, em memória)
  → Dado criptografado com chave efêmera
  → Chave destruída (sobrescrita com zeros, deletada)
  → Apenas context_snapshot é gravado

Fluxo de descriptografia (reconstrução):
context_snapshot recuperado do armazenamento
  → Oráculo verifica assinatura do snapshot
  → Setup (local, no dispositivo) + snapshot → KDF → mesma chave efêmera
  → Dado descriptografado
  → Chave destruída novamente

 
3. Living Wallet — Integração com BBP
A Living Wallet é a aplicação do protocolo Tecnologia Viva ao problema de segurança de carteiras blockchain. Ela resolve o maior problema da Web3 atual: quem tem a seed phrase tem os ativos. Na Living Wallet, ter a seed não é suficiente — é necessário o contexto completo do dispositivo original.

3.1 Diferença para wallets convencionais
Aspecto	Wallet Convencional	Living Wallet
Chave privada	Armazenada permanentemente	Nunca armazenada
Seed phrase roubada	Acesso total imediato	Inútil sem setup + oráculo
Wallet copiada para outro PC	Funciona normalmente	Modo stealth / dados embaralhados
Recuperação após perda	Seed phrase (frágil)	Snapshot + novo setup autenticado
Autodestruição em roubo	Não existe	Ativa automaticamente
Integração com DNA BBP	Não existe	DNA BBP como raiz de identidade

3.2 Integração com BBP
O protocolo BBP (Bitcoin Blueprint Protocol) permite inscrever o DNA de qualquer ativo no Bitcoin de forma compacta. A Living Wallet usa este DNA como componente da identidade criptográfica:
wallet_identity_root = SHA256(
    BBP_DNA_hash          # identidade determinística do ativo/usuário
  + device_setup_id       # identidade do dispositivo
  + oracle_context_hash   # momento do universo digital
)

Consequência direta: o DNA BBP passa a ser também um componente de recuperação. Se o usuário perder o dispositivo, o BBP DNA (armazenado na blockchain Bitcoin) é um dos ingredientes para reconstruir o setup em um novo dispositivo, combinado com autenticação biométrica e o oráculo.

3.3 Cenários de segurança
Cenário	O que acontece	Resultado
Ladrão copia arquivos da wallet	Sem setup do dispositivo original → chave não derivada	Modo stealth ativado
Seed phrase exposta	Sem context_snapshot válido → inutilizável	Ativos protegidos
Dispositivo perdido (dentro de 7 dias)	Snapshot ainda válido + nova autenticação	Recuperação possível
Dispositivo perdido (após 7 dias)	BBP DNA + biometria + novo setup	Recuperação via protocolo
Ataque de força bruta	Sem setup + snapshot + oráculo → impossível	Superfície de ataque zero
Vírus tenta abrir wallet	Sem setup válido do processo → oráculo rejeita	Acesso bloqueado

3.4 Código — Estrutura dos módulos
O código real da Living Wallet está implementado em Python e organizado nos seguintes módulos:
•	living_wallet.py — módulo principal com LiveSetupID, OracleClient, LivingKey, LivingWallet, LiveDataFractal
•	oracle_network.py — oráculo descentralizado com ShamirSharing, OracleNode, OracleCoordinator, OracleNetwork

Principais classes e responsabilidades:
Classe	Responsabilidade	Dado que armazena
LiveSetupID	Fingerprint do dispositivo + usuário	Apenas hashes — nenhum segredo em claro
OracleClient	Comunica com oráculo, deriva chaves	Nenhum — stateless
LivingKey	Encapsula chave efêmera com ciclo de vida	Chave em memória — destruída no __exit__
LivingWallet	Interface da wallet para o usuário	Apenas snapshots (sem chaves)
LiveDataFractal	Fragmentação determinística	Ordem derivada em runtime — não gravada
OracleNetwork	Rede descentralizada de oráculos	Shares individuais por nó — nunca o segredo completo

 
4. Oráculo Descentralizado — Threshold / MPC
O maior risco de um oráculo centralizado é ser um ponto único de falha: se comprometido, todos os snapshots ficam vulneráveis. A solução é distribuir o oráculo em N nós com threshold T, onde nenhum nó individual tem o segredo completo — e qualquer T nós são suficientes para operar.
Esta arquitetura preserva o espírito da Tecnologia Viva: a rede em si é o oráculo. Nenhuma entidade — nem o criador do protocolo — controla a entropia depois que a rede está viva.

4.1 Shamir Secret Sharing — a matemática por trás
O segredo mestre é dividido via Shamir Secret Sharing em campo finito GF(p). A propriedade fundamental é que qualquer subconjunto de T shares reconstrói o segredo, mas qualquer subconjunto de T-1 shares não revela absolutamente nada sobre ele — matematicamente impossível, não apenas computacionalmente difícil.
Segredo S dividido em 5 shares (threshold 3):

  Share 1: (1, y1)   → nó oracle-01
  Share 2: (2, y2)   → nó oracle-02
  Share 3: (3, y3)   → nó oracle-03
  Share 4: (4, y4)   → nó oracle-04
  Share 5: (5, y5)   → nó oracle-05

  oracle-01 + oracle-03 + oracle-05  → reconstrói S  ✅
  oracle-01 + oracle-02              → nada revelado ✅
Após a inicialização da rede, o segredo mestre S é destruído. Nenhum nó — e nenhuma pessoa — o conhece. A rede opera com shares distribuídas.

4.2 Multi-Party Computation (MPC) — sem revelar o segredo
Na geração de snapshots, os nós não precisam revelar suas shares. Cada nó contribui com um hash parcial derivado da sua share + o challenge público. O coordenador combina os parciais deterministicamente. Nenhum nó revela nada sobre sua share individual.
Protocolo por rodada:

  challenge = SHA256(setup_id + beacon + block_hash + event_data)

  Nó i computa:
    partial_i = SHA256(share_i || challenge || local_entropy_i)

  Coordenador combina:
    combined = SHA256(partial_1 || partial_2 || ... || partial_T)

  combined → context_snapshot → deriva chave efêmera

4.3 Propriedades de segurança
Propriedade	Descrição	Status
Threshold security	T-1 nós comprometidos não revelam nada	✅ Garantido matematicamente
Verificabilidade	Qualquer participante verifica o snapshot	✅ Assinaturas Ed25519 por nó
Resistência a nó malicioso	Nó mentiroso detectado via hash inconsistente	✅ Protocolo de verificação
Entropia emergente	Cada nó contribui entropia local independente	✅ Impossível de prever
Descentralização real	Nenhum nó controla a rede	✅ Sem ponto central de falha
Expansão viva	Novos nós entram com novas shares	✅ Protocolo de onboarding

4.4 Como a rede cresce — tecnologia viva em ação
Cada novo participante do protocolo (empresa, usuário, dispositivo) pode opcionalmente se tornar um nó do oráculo. Quando isso acontece:
•	A entropia total da rede aumenta — mais fontes de caos imprevisível.
•	O oráculo mestre se fragmenta mais — menor superfície de ataque por nó.
•	Os ciclos vivos mudam automaticamente com cada nova interação.
•	Nem o criador do protocolo consegue prever os dados que a rede usa para gerar entropia.

Isso é o que Léo chamou desde o início de 'tecnologia viva': um sistema que se fortalece com o uso, onde as pessoas são literalmente os componentes de segurança.

 
5. Fontes de Entropia e Entropy Poisoning
O maior risco específico desta arquitetura é o entropy poisoning: um atacante que consegue influenciar as fontes de entropia pode potencialmente tornar a geração de chaves previsível. A defesa é o entropy mixing — múltiplas fontes independentes, onde controlar uma não é suficiente.

5.1 Fontes de entropia e seus riscos
Fonte	Imprevisibilidade	Risco de manipulação	Solução
Randomness Beacon (drand)	Muito alta	Muito baixo — assinado por consórcio	Usar como âncora obrigatória
Block hash Bitcoin/Ethereum	Alta	Baixo — custo de mineração	Usar block_hash + height
Temperatura CPU	Média	Médio — pode ser controlada	Combinar com outras fontes
Jitter de rede	Alta	Baixo em rede real	Medição em múltiplos pontos
Movimento do mouse	Alta (humano)	Alto (pode ser simulado)	Hash de padrão temporal, não posição
Entropia de processo local	Alta	Médio	PID + timing de instruções
Interações da rede blockchain	Alta	Médio — requer custo econômico	Últimos N contratos, não apenas 1

5.2 Entropy Mixing — a defesa
O sistema não depende de nenhuma fonte isolada. A chave final é função de todas as fontes juntas. Se um atacante controla uma fonte, ainda precisa controlar todas as outras simultaneamente — o que tem custo proibitivo:
living_entropy = SHA256(
    drand_beacon_random     # imprevisível, verificável publicamente
  + btc_block_hash          # custo econômico para manipular
  + cpu_jitter_hash         # físico, local
  + setup_device_entropy    # vinculado ao hardware específico
  + network_timing_hash     # comportamento real da rede
)

O drand Randomness Beacon (drand.cloudflare.com) é uma rede de consórcio que gera aleatoriedade pública verificável a cada 30 segundos, assinada por BLS com threshold. É a âncora mais forte disponível hoje para entropy mixing em sistemas como este.

•	Auditoria de segurança externa do protocolo completo.

 
8. Glossário Técnico
Termo	Definição
Context Snapshot	Registro determinístico e assinado do estado do sistema no momento de geração de uma chave. Substitui a chave armazenada.
Chave Efêmera	Chave criptográfica que existe apenas em memória RAM durante seu uso. Destruída por sobrescrita imediatamente após.
LiveSetupID	Fingerprint único do dispositivo + usuário. Nunca sai do dispositivo — apenas referências (hashes) são compartilhadas.
Entropy Mixing	Técnica de combinar múltiplas fontes de entropia independentes, de modo que controlar uma não compromete o sistema.
Entropy Poisoning	Ataque onde um adversário injeta dados controláveis em fontes de entropia para tornar a geração de chaves previsível.
Shamir Secret Sharing	Esquema matemático que divide um segredo em N shares, das quais qualquer T reconstrói o segredo sem que T-1 shares revelem nada.
Threshold Cryptography	Criptografia onde operações exigem cooperação de T-de-N partes — nenhuma parte sozinha é suficiente.
MPC (Multi-Party Computation)	Protocolo onde múltiplas partes computam juntas uma função sem revelar suas entradas individuais.
drand	Rede de consórcio (Cloudflare + parceiros) que gera aleatoriedade pública, verificável e não manipulável a cada 30 segundos.
KDF (Key Derivation Function)	Função que transforma um segredo de entrada em material de chave criptográfica. Ex: HKDF-SHA256.
TPM (Trusted Platform Module)	Chip de hardware que armazena chaves de forma que não podem ser extraídas — clonagem física é inviável.
CRYSTALS-Kyber / Dilithium	Algoritmos pós-quânticos padronizados pelo NIST em 2024 — resistentes a ataques de computadores quânticos.
BBP DNA	Representação compacta (poucos bytes) de qualquer ativo digital no Bitcoin, usada como raiz de identidade na Living Wallet.
Modo Stealth	Comportamento da wallet quando o setup não é validado: mostra dados plausíveis mas falsos, sem revelar que o acesso foi negado.
Oráculo Vivo	Sistema que fornece entropia viva e context_snapshots. Pode ser um servidor, uma rede de nós, ou hardware físico (Olho de Deus).
