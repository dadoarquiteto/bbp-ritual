BBP-0014
Living Cryptography Layer
Camada de Criptografia Viva do Protocolo BBP
Integração da Tecnologia Viva com Inscrição Ordinal, Manifesto e Arweave
Status: Draft  |  Versão: 2.1  |  Autor: BBP Core Team
Complementa: BBP-0001, BBP-0002, BBP-0012, BBP-0015
Atualizado para Tecnologia Viva v3.0 — Abril 2026
 
Resumo
Este documento define a integração da Tecnologia Viva — um protocolo de criptografia contextual com chaves efêmeras — ao protocolo BBP. A camada opera como proteção opcional em três momentos críticos do ciclo de vida de um ativo BBP: a assinatura da inscrição Ordinal no Bitcoin, o armazenamento do manifesto e assets no Arweave, e a identidade criptográfica do criador ancorada no BBP:id:identity.

A versão 2.1 deste documento reflete as atualizações da Tecnologia Viva v3.0, incorporando entropia comportamental distribuída, hash chain viva alimentada pela **rede P2P BBPNetwork**, e SHA3-256 em todas as operações de hash. O campo quantum_resistant no manifesto passa a ser true em implementações que seguem BBP-0015.

**ATUALIZAÇÃO v2.1 (Abril 2026):** A camada P2P foi migrada de GUN.js para BBPNetwork (BroadcastChannel + WebRTC). A hash chain viva agora é alimentada pelo BBPNetwork em vez da implementação anterior. As referências à rede P2P neste documento refletem essa mudança.

Este documento é complementar ao BBP-0012 (Registro no Bitcoin e Armazenamento no Arweave) e ao BBP-0015 (Post-Quantum Security Standard). Leia BBP-0012 e BBP-0015 antes de implementar esta especificação.

1. Visão Geral
1.1 O problema que esta camada resolve
No fluxo atual do BBP-0012, o criador assina o contract_hash do manifesto com sua chave privada convencional (ECDSA). Essa chave existe permanentemente no dispositivo. Se for comprometida:
•	Qualquer ativo inscrito pelo criador tem sua autoria questionável.
•	A chave pode assinar inscrições fraudulentas em nome do criador.
•	Não há como provar que a assinatura ocorreu em momento e contexto específicos.
A Tecnologia Viva resolve os três problemas simultaneamente. A versão 2.0 adiciona um quarto: a rede que valida as assinaturas é alimentada pelo comportamento humano dos próprios usuários, tornando o estado do sistema irreproduzível por atacantes externos. **Na versão 2.1, esta rede é o BBPNetwork.**

1.2 O que a camada adiciona — v2.1
Aspecto	Sem BBP-0014	Com BBP-0014 v2.1
Chave privada do criador	Armazenada permanentemente	Nunca armazenada — gerada e destruída
Prova de autoria	Assinatura ECDSA convencional	Assinatura + context_snapshot verificável
Fonte de entropia	Ausente	Beacon drand + Bitcoin + entropia comportamental da **rede P2P BBPNetwork**
Hash chain viva	Ausente	Estado irreproduzível alimentado por cada interação humana na **BBPNetwork**
Recuperação de identidade	Seed phrase (fragil)	BBP DNA + oraculo + setup fisico
Proteção do manifesto no Arweave	Upload em claro	Fragmentação criptografada opcional
Resistência quântica	Não	Sim — zero curvas elipticas, SHA3-256, Dilithium3 (com BBP-0015)

1.3 Compatibilidade
A camada BBP-0014 é retrocompatível. Ativos registrados sem ela continuam válidos. Resolvers que não implementam BBP-0014 continuam funcionando — apenas não verificam o context_snapshot adicional. **A migração de GUN.js para BBPNetwork não afeta a compatibilidade.**

 
2. Integração com a Inscrição Ordinal (Bitcoin)
2.1 Fluxo modificado de assinatura — v2.1
O fluxo do BBP-0012 é estendido da seguinte forma:

Fluxo BBP-0014 v2.1 — Assinatura com chave efemera:  1. Template gera bbp_manifest_unsigned.json (sem alteracao) 2. Template calcula contract_hash com SHA3-256 (v2.0: era SHA-256) 3. [NOVO v2.0] BehavioralEntropyCollector captura entropia da interacao **via BBPNetwork** 4. [NOVO] Oraculo Vivo gera context_snapshot:    - beacon_round      : rodada atual do drand randomness beacon    - beacon_random     : aleatoriedade verificavel do beacon    - block_height      : altura atual do bloco Bitcoin    - block_hash_prefix : hash do bloco (prefixo 16 chars)    - event_hash        : SHA3-256(contract_hash + timestamp)  [v2.0: era SHA-256]    - timestamp_utc     : momento exato da operacao    - setup_id_ref      : referencia (prefixo) do LiveSetupID do criador    - genesis_seed      : ancora do usuario na hash chain viva  [NOVO v2.0]    - behavioral_contribution : hash da entropia comportamental  [NOVO v2.0] 5. [NOVO] Chave efemera derivada:    key = HKDF-SHA3-256(context_snapshot + setup_id_completo)  [v2.0: era HMAC-SHA256] 6. [NOVO] contract_hash assinado com Dilithium3  [v2.0: era ECDSA] 7. [NOVO] Chave efemera destruida (sobrescrita em memoria) 8. [NOVO] context_snapshot adicionado a inscricao como living_proof 9. Template gera bbp_manifest_signed.json (estendido)

2.2 Formato estendido da inscrição Ordinal — v2.1
O JSON da inscrição Ordinal recebe o campo opcional living_proof, atualizado para incluir genesis_seed e behavioral_contribution:

{   "p": "bbp",   "op": "identity",   "id": "BBP:id:art:fredo:2654",   "timestamp": 1700000000,   "creator": "bc1qxyz...",   "signature": "base64_assinatura_dilithium3",   "binding": "BBP:bind:arweave:abc123",   "living_proof": {     "version": "bbp-lc-v2",     "oracle_id": "fe9bccac764083da",     "beacon_round": 382947,     "beacon_random": "82b7c2f39ad55e1a9b3fd22c...",     "block_height": 901234,     "block_hash_prefix": "0000000000000000",     "event_hash": "SHA3-256_hash_aqui...",     "timestamp_utc": "2026-04-01T19:27:56Z",     "setup_id_ref": "639a195398d69dd1",     "genesis_seed": "ancora_na_hash_chain_viva...",     "behavioral_contribution": "hash_entropia_comportamental...",     "snapshot_sig": "1b38f0423f7bee62383c7a23..."   } }

2.3 O que o living_proof prova — v2.1
Campo	O que prova
beacon_round + beacon_random	A assinatura ocorreu apos aquela rodada do beacon — prova temporal verificavel por qualquer pessoa
block_height + block_hash_prefix	A assinatura ocorreu apos aquele bloco Bitcoin ser minerado — ancora temporal na blockchain
event_hash (SHA3-256)	Liga o living_proof ao contract_hash daquele manifesto. v2.0: SHA3-256 em vez de SHA-256
genesis_seed	NOVO v2.0 — ancora do usuario na hash chain viva do oraculo. Gravado no Arweave.
behavioral_contribution	NOVO v2.0 — prova que entropia humana real foi incorporada no momento da assinatura **via BBPNetwork**
snapshot_sig	O oraculo assinou este snapshot — garante que nao foi fabricado apos o fato
setup_id_ref	Liga a assinatura ao dispositivo especifico do criador — sem revelar o setup completo

2.4 Verificação por resolvers — v2.1
Verificacao do living_proof v2.1:  1. Extrair beacon_round do living_proof 2. Consultar drand: GET https://drand.cloudflare.com/public/{beacon_round} 3. Confirmar que beacon_random corresponde a resposta do drand 4. Confirmar que block_height e block_hash_prefix existem na blockchain Bitcoin 5. Recalcular event_hash = SHA3-256(contract_hash + timestamp_utc)  [v2.0: SHA3-256] 6. Confirmar que event_hash corresponde ao campo do living_proof 7. Verificar snapshot_sig com Dilithium3  [v2.0: era HMAC/ECDSA] 8. [NOVO v2.0] Verificar genesis_seed contra registro no Arweave do usuario  Se todos os passos passam: living_proof e valido. A assinatura ocorreu naquele momento, naquele contexto, naquele dispositivo, com entropia comportamental humana real incorporada **e transmitida via BBPNetwork**.

 
3. Integração com o Manifesto e Arweave
3.1 Campo security no manifesto — v2.1
O manifesto BBP-0002 recebe o campo opcional security atualizado para v2.1. O campo quantum_resistant passa a ser true com BBP-0015 implementado:

{   "bbp_version": "1.2",   "identity": "BBP:id:art:fredo:2654",   "standard": "nft",   "origin": { "network": "bitcoin", "reference": "inscription_847392" },   "security": {     "layer": "bbp-living-cryptography-v2",     "quantum_resistant": true,     "algorithms": {       "signature": "Dilithium3-ML-DSA-FIPS204",       "hash": "SHA3-256",       "kdf": "HKDF-SHA3-256"     },     "living_proof": {       "oracle_id": "fe9bccac764083da",       "beacon_round": 382947,       "genesis_seed": "ancora_hash_chain...",       "behavioral_contribution": "hash_entropia...",       "snapshot_sig": "1b38f0423f7bee62..."     },     "fragmented": false   } }

3.2 Papel do Arweave na arquitetura v2.1
Na versão 2.1, o Arweave tem um papel expandido além de armazenar manifestos. Ele é a memória permanente e auditável da hash chain viva:
•	Armazena manifestos BBP e assets (papel original — sem alteracao).
•	Armazena o genesis_seed de cada usuario — ancora imutavel de entrada na hash chain.
•	Armazena checkpoints periodicos da hash chain — auditoria publica verificavel por qualquer pessoa.
•	Qualquer resolver pode verificar a legitimidade de um genesis_seed consultando o Arweave sem confiar em nenhuma parte central.

Distinção importante: os nós do Arweave guardam e servem arquivos — não executam lógica nem contribuem entropia. A entropia viva vem da **rede P2P BBPNetwork**. O Arweave é a memória permanente; a **BBPNetwork** é o processamento vivo.

3.3 Proteção opcional dos assets no Arweave
Assets sensíveis podem ser fragmentados antes do upload usando o LiveDataFractal. O campo fragmented é true e o manifesto inclui o mapa de reconstrução:

"security": {   "layer": "bbp-living-cryptography-v2",   "quantum_resistant": true,   "fragmented": true,   "fragment_map": {     "n_fragments": 12,     "snapshot_ref": "SHA3-256_fingerprint...",     "genesis_seed_ref": "ancora_hash_chain...",     "reconstruction": "requires_setup_and_oracle_and_behavioral_chain"   },   "living_proof": { "..." } }

3.4 Fluxo completo de upload para Arweave — v2.1
Passo	Ação	v1.0 original	v2.1 atualizado
1	Gerar manifesto unsigned	Sem alteracao	Sem alteracao
2	Calcular contract_hash	SHA-256	SHA3-256
3	Capturar entropia comportamental	—	BehavioralEntropyCollector registra interacao do usuario **via BBPNetwork**
4	Gerar context_snapshot	Oraculo gera snapshot	Oraculo gera snapshot com genesis_seed e behavioral_contribution
5	Derivar chave efemera	HMAC-SHA256	HKDF-SHA3-256
6	Assinar contract_hash	ECDSA	Dilithium3 (com BBP-0015)
7	Fragmentar assets (opcional)	LiveDataFractal	LiveDataFractal com seed SHA3-256
8	Upload manifesto + security	quantum_resistant: false	quantum_resistant: true, genesis_seed no Arweave
9-10	Inscricao Bitcoin	living_proof v1	living_proof v2 com genesis_seed e behavioral

 
4. Identidade Criptográfica via BBP:id:identity
4.1 O BBP DNA como raiz de identidade
Com BBP-0014 v2.1, o DNA do ativo de identidade do criador funciona como IdentityRoot. A derivação usa SHA3-256 em vez de SHA-256:

Criador registra seu ativo de identidade:   BBP:id:identity:global:leo_ramalho  IdentityRoot com SHA3-256 (v2.0):   identity_root = SHA3-256(BBP:id:identity:global:leo_ramalho)  Todas as chaves efemeras do criador derivam de:   key = HKDF-SHA3-256(identity_root + device_setup + oracle_snapshot + genesis_seed)  Consequencia: qualquer ativo assinado pelo criador pode ser verificado contra o IdentityRoot registrado no Bitcoin.

4.2 Fluxo de recuperação de identidade — v2.1
Recuperacao de identidade BBP-0014 v2.1:  1. Criador apresenta: BBP:id:identity:global:leo_ramalho 2. Resolver confirma que o DNA existe no Bitcoin (imutavel) 3. Criador apresenta genesis_seed registrado no Arweave 4. Criador se autentica com biometria ou backup secreto 5. Oraculo valida genesis_seed contra hash chain atual **da BBPNetwork** 6. Novo context_snapshot gerado para o novo dispositivo 7. Novo LiveSetupID gerado no novo dispositivo 8. identity_root reutilizado — chaves efemeras futuras derivam do mesmo root 9. Acesso restaurado — nenhuma seed phrase envolvida

5. Entropia Comportamental — Novo em v2.0, mantido em v2.1
A versão 2.0 introduziu o BehavioralEntropyCollector como componente central do protocolo. **Na versão 2.1, ele opera sobre a BBPNetwork em vez da implementação P2P anterior.** Cada interação humana na rede P2P do BBP alimenta a hash chain do oráculo, tornando a rede viva por natureza.

5.1 Como a entropia comportamental entra no living_proof
•	Usuario interage com a **BBPNetwork** (clique, conexao de carteira, revelacao de ativo).
•	BehavioralEntropyCollector captura o padrao matematico da interacao — nunca o dado em si.
•	Hash calculado: SHA3-256(timing + tipo_evento + timestamp_ns + beacon_atual).
•	Hash contribui para o context_snapshot como behavioral_contribution.
•	Hash avanca a hash chain do oraculo via ratcheting irreversivel.
•	O genesis_seed do usuario e derivado do estado atual da chain no momento da operacao.

5.2 Privacidade — o que é coletado e o que não é
O que é coletado	O que não é coletado	Como é processado
Tempo entre cliques (ms)	O que o usuario clicou	SHA3-256(timing + contexto) → hash anonimo
Temperatura da CPU no momento	Dados pessoais do usuario	Incorporado ao hash do evento
Tipo do evento (click, reveal, tx)	Conteudo da interacao	String anonima no hash
Jitter de rede local	Endereco IP ou localizacao	Valor numerico no hash

O dado original é descartado imediatamente após o hash ser calculado. O que alimenta a chain é o hash — não o comportamento. Isso garante que o sistema nunca armazena nem processa dados pessoais dos usuários.

6.2 Exemplo mínimo de integração — v2.1
**from bbp_network import BBPNetwork**
from living_wallet_v3 import (     BehavioralEntropyCollector, LiveSetupID, OracleClient, LivingKey )
import json, hashlib

**# 0. Inicializar BBPNetwork (substitui GUN.js)**
**network = BBPNetwork()**
**network.init()**

# 1. Coletor de entropia comportamental
behavioral = BehavioralEntropyCollector(**network=network**)
behavioral.collect_event('page_load')

# 2. Setup do criador
setup = LiveSetupID(
    user_id='leo_ramalho',
    secret_pin='seu_pin',
    behavioral_collector=behavioral
)

# 3. Manifesto BBP
manifest = { ... }
manifest_bytes = json.dumps(manifest, sort_keys=True).encode()

# 4. contract_hash com SHA3-256 (pos-quantico)
contract_hash = hashlib.sha3_256(manifest_bytes).digest()

# 5. Oraculo gera context_snapshot com genesis_seed e behavioral
oracle = OracleClient(oracle_url='https://oracle.bbp.io')
snapshot = oracle.get_context_snapshot(setup, manifest_bytes, behavioral)

# 6. Chave efemera — existe apenas durante a assinatura
key_bytes = oracle.reconstruct_key_from_snapshot(snapshot, setup)
with LivingKey(key_bytes, snapshot) as living_key:
    signature = living_key.sign(contract_hash) # Chave destruida aqui

# 7. Montar inscricao com living_proof v2
inscription = {
    'p': 'bbp', 'op': 'identity',
    'id': 'BBP:id:art:fredo:2654',
    'signature': signature.hex(),
    'living_proof': snapshot.__dict__
}
